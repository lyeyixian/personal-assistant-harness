# Research: Python Agent SDK Comparison

**Ticket:** [#2](https://github.com/lyeyixian/personal-assistant-harness/issues/2) · **Date:** 2026-07-15 · **Scope:** Claude Agent SDK (Python), OpenAI Agents SDK, Google ADK, Pydantic AI, plus smolagents as a lightweight reference point.

All claims are sourced from primary sources only: official docs, GitHub repos/source/releases, PyPI, pypistats.org, and first-party vendor blog posts. GitHub/PyPI numbers were fetched live from `api.github.com` and `pypistats.org` on 2026-07-15. This document surfaces facts and trade-offs; the SDK pick is a separate decision ticket (#3).

## Per-SDK notes

### Claude Agent SDK (Python)

- **What it is:** the full Claude Code engine (agent loop, context management, built-in tools, permissions, hooks) packaged as a self-hosted library. Each `query()` spawns a bundled `claude` CLI subprocess owned by the SDK — no separate CLI install needed, but you are running an agent *harness*, not making raw API calls ([hosting docs](https://code.claude.com/docs/en/agent-sdk/hosting.md)).
- **Model flexibility:** Claude models only, but reachable through five backends: Anthropic API, Amazon Bedrock, Vertex AI, Azure Foundry, and Claude on AWS ([overview](https://code.claude.com/docs/en/agent-sdk/overview.md)). This is multi-cloud, not multi-provider — you cannot point it at GPT or Gemini.
- **Unique strengths:** built-in production tools (Read/Write/Edit, Bash, Glob/Grep, WebSearch/WebFetch), subagents with isolated contexts, automatic context compaction with `compact_boundary` events, and the most granular permission system of the group: `allowed_tools` patterns like `Bash(ls *)`, six permission modes (`default`/`acceptEdits`/`plan`/`dontAsk`/`bypassPermissions`/`auto`), and hooks (`PreToolUse`, `PostToolUse`, `PreCompact`, `SubagentStart/Stop`, ...) that run in your process and can deny even under `bypassPermissions` ([permissions](https://code.claude.com/docs/en/agent-sdk/permissions.md), [hooks](https://code.claude.com/docs/en/agent-sdk/hooks.md)).
- **Sessions:** continue/resume/fork by session ID; JSONL transcripts under `~/.claude/projects/`; `SessionStore` adapters for S3/Redis/Postgres; CLAUDE.md as persistent memory ([sessions](https://code.claude.com/docs/en/agent-sdk/sessions.md)).
- **Maturity:** created 2025-06-11 (renamed from claude-code-sdk); 125 releases, still 0.2.x. Very active (release the day before this research) ([repo](https://github.com/anthropics/claude-agent-sdk-python), [CHANGELOG](https://github.com/anthropics/claude-agent-sdk-python/blob/main/CHANGELOG.md)).
- **Idiomaticity:** async-first (anyio), dataclass message types, plain-dict tool schemas auto-converted to JSON Schema. Fewer generics than Pydantic AI; simpler but less type-safe. Feature parity with the TS SDK is close but not exact (e.g., `SessionStart`/`SessionEnd` hooks and `streamInput()` are TS-only) ([Python ref](https://code.claude.com/docs/en/agent-sdk/python.md), [streaming modes](https://code.claude.com/docs/en/agent-sdk/streaming-vs-single-mode.md)).
- **Resource model:** Anthropic's own hosting baseline is ~1 GiB RAM / 5 GiB disk / 1 CPU per agent subprocess — heavier than a pure-library SDK ([hosting](https://code.claude.com/docs/en/agent-sdk/hosting.md)).

### OpenAI Agents SDK (Python)

- **What it is:** a lightweight multi-agent framework — the "production-ready upgrade" of the Swarm experiment — launched 2025-03-11 alongside the Responses API ([docs intro](https://openai.github.io/openai-agents-python/), [announcement](https://openai.com/index/new-tools-for-building-agents/)).
- **Model flexibility:** described as provider-agnostic ("OpenAI Responses and Chat Completions APIs, as well as 100+ other LLMs") via LiteLLM (beta) or the newer `any-llm` extra, plus custom `ModelProvider` ([models](https://openai.github.io/openai-agents-python/models/), [PyPI](https://pypi.org/project/openai-agents/)). Caveats: hosted tools (web search, file search, code interpreter), hosted MCP, and server-side conversation state (`conversation_id`, `previous_response_id`) work only with OpenAI's Responses API; tracing uploads to OpenAI by default and errors without an OpenAI key unless disabled ([models](https://openai.github.io/openai-agents-python/models/), [tools](https://openai.github.io/openai-agents-python/tools/)).
- **Primitives:** `Agent` (generic over a context type), `Runner.run/run_sync/run_streamed`, `@function_tool` (schema from type hints via Pydantic, description from docstring), handoffs (`transfer_to_<agent>` tools with typed input and history filters), input/output/tool guardrails, `agent.as_tool()` ([agents](https://openai.github.io/openai-agents-python/agents/), [handoffs](https://openai.github.io/openai-agents-python/handoffs/), [guardrails](https://openai.github.io/openai-agents-python/guardrails/)).
- **Sessions:** the widest set of drop-in backends — SQLite (default), SQLAlchemy, Redis, MongoDB, Dapr, OpenAI Conversations — plus `EncryptedSession` and an auto-compaction wrapper. No built-in semantic/long-term memory ([sessions](https://openai.github.io/openai-agents-python/sessions/)).
- **HITL:** `needs_approval` on tools pauses the run; `RunState.to_json()` lets you serialize a paused run for long-lived approval workflows ([HITL](https://openai.github.io/openai-agents-python/human_in_the_loop/)). First-party Temporal integration for durable execution ([Temporal blog](https://temporal.io/blog/announcing-openai-agents-sdk-integration), [pyproject extras](https://github.com/openai/openai-agents-python/blob/main/pyproject.toml)).
- **Maturity:** still 0.x after ~16 months, releasing every 1–2 weeks — breaking changes remain possible; pin versions ([releases](https://github.com/openai/openai-agents-python/releases)).
- **Idiomaticity:** Pydantic v2 everywhere (the closest Python analog to Zod), decorator tools, async-first with a `run_sync` convenience. A first-party TypeScript twin (`@openai/agents`) with near-identical primitives is a useful Rosetta stone for a TS dev ([TS docs](https://openai.github.io/openai-agents-js/)).

### Google ADK (Python)

- **What it is:** Google's agent framework, launched at Cloud NEXT on 2025-04-09; it is "the same framework powering agents within Google products like Agentspace" ([launch blog](https://developers.googleblog.com/en/agent-development-kit-easy-to-build-multi-agent-applications/)). v1.0 was declared production-ready at I/O 2025 ([I/O blog](https://developers.googleblog.com/agents-adk-agent-engine-a2a-enhancements-google-io/)); v2.0 GA landed 2026-05-19 with a new graph-based Workflow Runtime and Task API ([v2.0.0 release](https://github.com/google/adk-python/releases/tag/v2.0.0), ["Why we built ADK 2.0"](https://developers.googleblog.com/why-we-built-adk-20/)).
- **Model flexibility:** Gemini via registry strings; everything else via wrapper classes — `LiteLlm` for 100+ providers, and a native `AnthropicLlm` exists in source ([models](https://adk.dev/agents/models/), [anthropic_llm.py](https://github.com/google/adk-python/blob/main/src/google/adk/models/anthropic_llm.py)). Built-in tools (google_search, code execution) and live bidi streaming are Gemini-only ([limitations](https://adk.dev/tools/limitations/), [streaming](https://adk.dev/streaming/)).
- **Primitives:** `LlmAgent`, deterministic workflow agents (`SequentialAgent`/`ParallelAgent`/`LoopAgent`), `FunctionTool` from type hints + docstrings, `AgentTool`, sub-agent hierarchies; v2 adds `Workflow(edges=[...])` graphs with fan-out/fan-in, retry, and HITL nodes ([agents](https://adk.dev/agents/), [README](https://github.com/google/adk-python)).
- **Memory:** the richest managed-memory story — Session/State/Memory separation, state prefixes (`user:`, `app:`, `temp:`), and Vertex-backed `VertexAiMemoryBankService` (LLM-extracted memories) and RAG memory ([sessions](https://adk.dev/sessions/), [memory](https://adk.dev/sessions/memory/)) — but the good implementations are Google Cloud paid services; in-memory versions are explicitly dev-only.
- **Streaming:** unique bidi audio/video via Gemini Live API (`run_live()`), still marked Experimental as of July 2026 ([streaming](https://adk.dev/streaming/)).
- **HITL:** `require_confirmation` on tools is experimental and documented as unsupported with `DatabaseSessionService`/`VertexAiSessionService`; six before/after callbacks plus a plugin system ([confirmation](https://adk.dev/tools-custom/confirmation/), [callbacks](https://adk.dev/callbacks/), [plugins](https://adk.dev/plugins/)).
- **Ecosystem:** Python, TypeScript, Go, Java, Kotlin/Android SDKs; A2A protocol; 70+ official samples in [adk-samples](https://github.com/google/adk-samples); managed deployment target renamed Agent Engine → "Agent Runtime" under the Gemini Enterprise Agent Platform ([deploy](https://adk.dev/deploy/)).
- **Idiomaticity:** class/config-heavy declarative construction (closer to configuring NestJS than writing Express handlers), Pydantic as a hard dependency, fully async services since v1.0 ([v1.0.0 release](https://github.com/google/adk-python/releases/tag/v1.0.0)). Largest conceptual surface area of the four.

### Pydantic AI

- **What it is:** an agent framework from the Pydantic team with a stated goal of bringing "that FastAPI feeling" to GenAI development ([overview](https://pydantic.dev/docs/ai/overview/)). Repo created 2024-06-21; public launch late 2024; v1.0 on 2025-09-04 with 15M downloads during the pre-release period; v2.0 on 2026-06-23 ([v1 blog](https://pydantic.dev/articles/pydantic-ai-v1), [releases](https://github.com/pydantic/pydantic-ai/releases)).
- **Model flexibility:** the most provider-neutral of the four by design — first-class model classes for OpenAI, Anthropic, Gemini, xAI, Bedrock, Groq, Mistral, Cohere, HF and more, plus any OpenAI-compatible endpoint (Ollama, DeepSeek, Together, ...), with `FallbackModel` for automatic failover and `TestModel`/`FunctionModel` for offline testing ([models](https://pydantic.dev/docs/ai/models/overview/)).
- **Primitives:** `Agent[DepsType, OutputType]` — generics checked by mypy/pyright; `@agent.tool` with typed `RunContext` dependency injection; structured outputs where `result.output` *is* the declared type; streaming with partial Pydantic validation on every chunk; `iter()` and pydantic-graph for explicit state machines ([agent](https://pydantic.dev/docs/ai/core-concepts/agent/), [output](https://pydantic.dev/docs/ai/output/), [graph](https://pydantic.dev/docs/ai/graph/)).
- **Memory:** `message_history` primitives + JSON serialization; no built-in store — persistence is DIY, or you adopt one of four co-maintained durable-execution integrations (Temporal, DBOS, Prefect, Restate) ([message history](https://pydantic.dev/docs/ai/core-concepts/message-history/), [durable execution](https://pydantic.dev/docs/ai/integrations/durable_execution/overview/)).
- **HITL:** `requires_approval=True` / `ApprovalRequired`, deferred tools for external execution, resumable via `DeferredToolResults` — shipped in v1.0 ([deferred tools](https://pydantic.dev/docs/ai/tools-toolsets/deferred-tools/), [v1.0.0](https://github.com/pydantic/pydantic-ai/releases/tag/v1.0.0)).
- **MCP:** client (stdio, streamable HTTP, deprecated SSE) and server via FastMCP, including MCP sampling and elicitation ([MCP client](https://pydantic.dev/docs/ai/mcp/client/), [server](https://pydantic.dev/docs/ai/mcp/server/)).
- **Maturity/backing:** MIT, backed by the Pydantic company, integrated (optionally) with Logfire/OpenTelemetry; 474 contributors — highest of the group; docs examples are unit-tested and the project claims 100% test coverage ([v1 blog](https://pydantic.dev/articles/pydantic-ai-v1)). Near-daily releases; v1.x still maintained alongside v2.
- **Idiomaticity:** the strongest match for a TS developer who values types — generics, write-time type errors, DI instead of globals, decorators-as-routes, and both sync and async APIs ([agent docs](https://pydantic.dev/docs/ai/core-concepts/agent/)).

### smolagents (brief)

- Hugging Face's "barebones library for agents that think in code": core logic in ~1,000 lines, `CodeAgent` writes Python as its actions (README claims ~30% fewer steps than JSON tool calling), `ToolCallingAgent` for standard JSON calls ([docs](https://huggingface.co/docs/smolagents/index), [README](https://github.com/huggingface/smolagents/blob/main/README.md), [guided tour](https://huggingface.co/docs/smolagents/guided_tour)).
- The API reference carries an explicit disclaimer: "Smolagents is an experimental API which is subject to change at any time" ([reference](https://huggingface.co/docs/smolagents/reference/agents)). The local Python executor is documented as *not* a security boundary; real isolation requires E2B/Docker/Modal executors ([secure code execution](https://huggingface.co/docs/smolagents/tutorials/secure_code_execution)).
- As of mid-2026 it is maintained but slowing: no release since v1.26.0 (2026-05-29), recent commits are mostly maintenance by a single HF maintainer ([commits](https://github.com/huggingface/smolagents/commits/main)). Downloads (~0.6M/month) are ~2 orders of magnitude below the other four. Relevant here mainly as a learning reference or for the code-as-actions pattern, not as the harness backbone.

## Cross-cutting trade-offs

- **Harness vs. framework vs. library.** Claude Agent SDK ships a complete opinionated harness (tools, permissions, compaction included — but a subprocess with ~1 GiB baseline). OpenAI Agents SDK and Pydantic AI are libraries where you assemble the loop from primitives. ADK is a full framework with a CLI, dev UI, eval, and a deployment product attached.
- **Lock-in is graded, not binary.** Every "multi-provider" claim has an asterisk: OpenAI's hosted tools/tracing/server-side sessions are OpenAI-only; ADK's built-in tools, live streaming, and managed memory are Gemini/Google-Cloud-only; Claude Agent SDK is Claude-only outright. Pydantic AI has the fewest provider-conditional features; smolagents similarly, at the cost of fewer features overall.
- **Versioning risk.** Claude Agent SDK and OpenAI Agents SDK are still pre-1.0 (0.2.x / 0.18.x) with weekly-ish releases. Pydantic AI and ADK both went through a v1→v2 break in mid-2026 and maintain parallel v1 lines — meaning much 2025-era tutorial content is stale for all four. Pinning versions is mandatory everywhere.
- **Portfolio signal.** By adoption numbers (stars, downloads, contributors), OpenAI Agents SDK leads, with ADK and Pydantic AI close behind; all four are first-party SDKs from major AI vendors (plus Pydantic), so any of them is a recognizable line on a portfolio. Claude Agent SDK has the smallest star count (7.6k) but 22M monthly downloads — heavy real-world use relative to its social footprint. (Numbers above; no first-party source ranks "employer recognition" — that part is inherently judgment.)
- **For a dev learning Python (from TS):** Pydantic AI and OpenAI Agents SDK teach transferable modern-Python idioms (Pydantic ≈ Zod, generics, decorators, asyncio). Claude Agent SDK has the smallest API to learn but teaches you *its* harness more than Python. ADK has the steepest conceptual load (agents + workflows + sessions + state prefixes + callbacks + plugins + extras).

## What the docs don't tell you

Gaps and gotchas found in primary sources but easy to miss:

- **Claude Agent SDK:** cost figures (`total_cost_usd`) are client-side estimates, not billing truth ([cost tracking](https://code.claude.com/docs/en/agent-sdk/cost-tracking.md)); no built-in session timeout (only `max_turns`/`max_budget_usd`); multi-tenant isolation requires explicitly disabling filesystem settings loading; streaming events don't include subagent deltas ([hosting](https://code.claude.com/docs/en/agent-sdk/hosting.md), [streaming](https://code.claude.com/docs/en/agent-sdk/streaming-output.md)). Python lags TS on a few features (input streaming, some hooks).
- **OpenAI Agents SDK:** tracing silently phones home to OpenAI even on non-OpenAI providers until disabled; session persistence only completes after you fully consume `stream_events()`; input guardrails run only on the *first* agent and output guardrails only on the *last*; LiteLLM adapter is beta and can drop usage metrics ([models](https://openai.github.io/openai-agents-python/models/), [streaming](https://openai.github.io/openai-agents-python/streaming/), [guardrails](https://openai.github.io/openai-agents-python/guardrails/)).
- **Google ADK:** the built-in-tool exclusivity rule (one built-in tool, alone, per agent, not in sub-agents) is a common architectural trap ([limitations](https://adk.dev/tools/limitations/)); tool confirmation doesn't work with the database/Vertex session services; MCP/A2A/LiteLLM/Anthropic are all optional extras missing from a bare install ([pyproject](https://github.com/google/adk-python/blob/main/pyproject.toml)); v2 session data is incompatible with pre-1.28 versions ([README](https://github.com/google/adk-python)); Google Cloud product renames (Agent Engine→Agent Runtime, Vertex RAG→Knowledge Engine) make older docs misleading; docs warn LiteLLM 1.82.7/1.82.8 contained unauthorized code ([litellm page](https://adk.dev/agents/models/litellm/)).
- **Pydantic AI:** v2 (June 2026) restructured the API into "capabilities", so v1-era tutorials don't match current docs; the docs site migrated from ai.pydantic.dev to pydantic.dev/docs/ai and deep links rot; the `mcp-run-python` sandbox repo was archived 2026-01-30 while still referenced in docs ([archived repo](https://github.com/pydantic/mcp-run-python)); no built-in persistence — you own storage.
- **General:** all five require Python ≥3.10; all move fast enough that any tutorial older than ~6 months should be distrusted; download counts include CI installs and aren't normalized.
